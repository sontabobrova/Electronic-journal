# Приемочные сценарии

Актуализировано: 23.05.2026.

## 1. Базовый запуск

Предусловия:

- выполнен `docker compose up --build -d`;
- сервисы `frontend`, `backend`, `db`, `redis`, `celery-worker`, `celery-beat` запущены.

Проверки:

- `GET http://localhost:8000/health/` возвращает `{"status":"ok"}`;
- `http://localhost:3000/` открывает frontend;
- `docker compose ps` показывает запущенные контейнеры;
- команда `docker compose exec backend python manage.py create_demo_data` завершается успешно;
- повторный запуск `create_demo_data` не создает дубликаты ключевых демо-записей.
- команда `docker compose exec backend python manage.py create_demo_data --reset --full` создает большой воспроизводимый демонстрационный набор;
- повторный запуск `docker compose exec backend python manage.py create_demo_data --full` не создает дубликаты.

## 2. Авторизация и роли

Проверки:

- администратор входит с `admin / admin123`;
- преподаватель входит с `teacher / teacher123`;
- студент входит с `student / student123`;
- после входа меню соответствует роли;
- после выхода приватные страницы недоступны;
- после трех неверных попыток входа пользователь временно блокируется;
- студент и преподаватель не получают доступ к административным разделам.

## 3. Кабинет администратора: пользователи

Проверки:

- администратор открывает `/admin`;
- во вкладке пользователей отображается список пользователей;
- администратор создает пользователя с ролью `Преподаватель`;
- в форме появляются поля профиля преподавателя;
- после сохранения преподаватель доступен в списке назначений;
- администратор создает пользователя с ролью `Студент`;
- в форме появляются поля профиля студента: номер, группа, дата зачисления;
- после сохранения студент доступен в списке студентов назначенной группы;
- администратор редактирует пользователя;
- администратор сбрасывает пароль;
- отключенный пользователь не должен входить в систему.

## 4. Кабинет администратора: учебный процесс

Проверки:

- создается группа;
- создается дисциплина;
- создается учебный период;
- создается назначение преподавателя на группу, дисциплину и период;
- созданное назначение отображается в таблице;
- назначение можно изменить;
- назначение можно удалить, если оно не используется связанными записями;
- группу, дисциплину и период можно редактировать;
- удаление используемых записей либо запрещается backend, либо требует предварительного удаления связанных записей.

## 5. Кабинет преподавателя

Проверки:

- преподаватель открывает `/teacher`;
- видит профиль и только свои назначения;
- видит студентов назначенных групп;
- создает работу журнала;
- изменяет работу журнала;
- удаляет работу журнала, если она не используется оценками;
- выставляет оценку студенту своей группы;
- изменяет оценку;
- удаляет оценку;
- создает занятие;
- изменяет занятие;
- удаляет занятие, если оно не используется посещаемостью;
- отмечает посещаемость;
- изменяет посещаемость;
- удаляет запись посещаемости;
- не может работать с чужими назначениями.

## 6. Кабинет студента

Проверки:

- студент открывает `/student`;
- видит свой профиль;
- видит только свои оценки;
- видит только свою посещаемость;
- видит средний балл и сводку посещаемости;
- не видит кабинет администратора;
- не видит кабинет преподавателя;
- не может изменять оценки, занятия и посещаемость.

## 7. Отчеты

Проверки:

- администратор открывает `/reports`;
- формирует отчет по успеваемости в `csv`;
- формирует отчет по успеваемости в `xlsx`;
- формирует отчет по успеваемости в `pdf`;
- формирует отчет по посещаемости в `csv`;
- формирует отчет по посещаемости в `xlsx`;
- формирует отчет по посещаемости в `pdf`;
- сформированный отчет имеет `download_url`;
- преподаватель формирует отчеты только по своим назначениям;
- студент не получает доступ к разделу отчетов.

## 8. Уведомления

Проверки:

- администратор или преподаватель открывает `/notifications`;
- видит только уведомления, адресованные текущему пользователю;
- отметка `прочитано` применяется только к уведомлению текущего пользователя;
- прочтение уведомления администратором не меняет уведомление преподавателя;
- задача `send_session_closing_reminders` создает напоминание преподавателю за 3 дня до окончания активного периода;
- повторный запуск задачи не создает дубликат одного и того же напоминания.

## 9. Аудит и безопасность

Проверки:

- все API, кроме healthcheck и login, требуют авторизации;
- попытки доступа не по роли возвращают `403`;
- в ответах есть `X-Request-ID`;
- в аудите фиксируются входы, ошибки входа, выходы, изменения пользователей, оценки, посещаемость и отчеты;
- в аудите отображаются пользователь, действие, объект и request id.

## 10. Frontend-полировка

Проверки:

- профильная информация пользователя находится в соответствующем кабинете;
- верхняя навигация не ломается на узких экранах;
- таблицы прокручиваются горизонтально при нехватке ширины;
- модальные формы не перекрывают критичные элементы интерфейса.
